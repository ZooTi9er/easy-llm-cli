# 自定义 API 配置问题修复项目需求说明文档 (PRD)

## 1. 项目概述

### 1.1 项目背景
用户报告 easy-llm-cli 的自定义 API 配置存在问题，怀疑程序调用的 API 不是从 `.env` 文件读取的，而是硬编码在源程序中。

### 1.2 项目目标
修复 easy-llm-cli 的自定义 API 配置读取机制，确保 `.env` 文件中的配置能够被正确加载和应用。

### 1.3 项目范围
- 修复 `.env` 文件加载机制
- 实现环境变量正确读取
- 验证自定义 API 功能正常工作
- 创建配置验证和调试机制

## 2. 问题根本原因分析

### 2.1 直接问题
1. **配置文件未被加载**: `.gemini/.env` 文件存在但可能未被正确加载到 `process.env`
2. **环境变量读取机制**: 代码直接读取 `process.env` 但配置文件可能未被加载
3. **入口点配置问题**: 缺少在应用启动时加载 `.env` 文件的机制

### 2.2 根本原因
- **缺少 dotenv 配置**: 虽然安装了 `dotenv` 包，但未在正确的入口点调用 `dotenv.config()`
- **入口点问题**: 真正的入口点是 `packages/cli/index.ts`，但缺少 `.env` 文件加载逻辑
- **配置验证缺失**: 没有配置验证和调试机制

### 2.3 核心问题链
**缺少 .env 文件加载 → 环境变量未被设置 → 自定义 API 配置失效 → 使用默认配置或硬编码值**

## 3. 解决方案设计

### 3.1 核心策略
1. **在入口点加载 .env 文件**: 在 `packages/cli/index.ts` 中添加 `dotenv.config()` 调用
2. **实现配置验证**: 创建配置验证模块确保配置正确加载
3. **添加调试机制**: 提供配置加载状态的调试信息
4. **改进错误处理**: 提供清晰的配置错误提示

### 3.2 详细解决方案

#### 步骤 1: 修改入口点文件
- 在 `packages/cli/index.ts` 顶部添加 `dotenv.config()` 调用
- 确保在应用启动时加载 `.env` 文件
- 添加调试日志验证配置加载

#### 步骤 2: 创建配置验证模块
- 在 `packages/core/src/config/` 中创建配置验证模块
- 使用 Zod 进行配置类型验证
- 提供默认值和错误提示

#### 步骤 3: 更新配置读取逻辑
- 修改 `packages/core/src/config/config.ts` 中的 `getModel()` 方法
- 修改 `packages/core/src/custom_llm/index.ts` 中的配置读取
- 确保使用验证后的配置对象

#### 步骤 4: 添加调试和验证机制
- 提供配置状态检查命令
- 添加详细的调试日志
- 实现配置验证功能

## 7. 风险评估和应对措施

### 7.1 风险评估矩阵

| 风险点 | 描述 | 发生概率 | 影响 | 应对措施 |
|--------|------|----------|------|----------|
| R1: 路径解析错误 | .env 文件路径不正确 | 中 | 高 | 使用绝对路径，添加路径验证 |
| R2: 配置覆盖问题 | 环境变量覆盖顺序问题 | 低 | 中 | 明确配置优先级，使用 override: true |
| R3: 依赖缺失 | zod 包未安装 | 低 | 中 | 检查并安装必要依赖 |
| R4: 构建问题 | 修改后构建失败 | 中 | 高 | 测试构建流程，备份原文件 |

### 7.2 主要风险应对策略

#### 风险1: 路径解析错误
- **绝对路径**: 使用 `path.resolve()` 确保路径正确
- **路径验证**: 添加路径存在性检查
- **调试日志**: 显示实际加载的文件路径

#### 风险2: 配置覆盖问题
- **override: true**: 确保 .env 文件覆盖系统环境变量
- **明确优先级**: 文档化配置优先级规则
- **配置验证**: 验证最终配置值
